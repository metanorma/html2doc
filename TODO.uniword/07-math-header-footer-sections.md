# Phase 7: Math, Headers/Footers, and Section Management

## Goal
Handle three remaining conversion challenges: OOXML math (already generated by html2doc), headers/footers (separate XML parts in DOCX), and multi-section document layout (page breaks with different headers/footers per section).

## Background

### Math
html2doc's `math.rb` already converts MathML to OOXML via Plurimath. The output is OMML XML embedded inline in the HTML document as `<m:oMath>` / `<m:oMathPara>` elements. For DOCX, we need to:
- Parse these embedded OMML elements into Uniword's math model objects
- Or pass the raw OMML XML through (Uniword can handle both approaches)

### Headers/Footers
html2doc's `process_header` reads a `header.html` file containing header/footer HTML with IE conditional comments. For DOCX:
- Headers/footers are separate XML parts: `word/header1.xml`, `word/footer1.xml`
- They contain `Paragraph` and `Table` objects (same body model)
- Section properties reference them via `HeaderReference`/`FooterReference` with rId

### Sections
The ISO fixture has 3 sections:
1. Cover page (even/default header+footer)
2. Copyright/front matter (even/default header+footer)
3. Body content (even/default/first header+footer, page numbering starts at 1)

## Tasks

### Math
- [ ] Decide math handling approach:
  - **Option A: Parse OMML into Uniword model objects** — iterate `<m:oMath>` elements, use `Uniword::Math::PlurimathAdapter.from_omml(xml)` to create `MathEquation` objects, store in `Paragraph.o_math_paras`
  - **Option B: Pass raw OMML XML** — inject the OMML XML string directly into the paragraph's raw XML during serialization
  - **Recommend: Option B initially** — faster to implement, and PlurimathAdapter has known limitations. Option A can be a follow-up for full model fidelity.

- [ ] Implement `DocxConverter#convert_math(element)`:
  - Detect `<m:oMath>` and `<m:oMathPara>` elements in the Nokogiri document
  - For inline math: wrap in a `Run` with raw OMML content
  - For block math (`oMathPara`): create a dedicated `Paragraph` with `oMathPara` child
  - Preserve alignment (`m:jc` values) and centering behavior from html2doc

- [ ] Handle math namespaces:
  - OOXML math namespace: `http://schemas.openxmlformats.org/officeDocument/2006/math`
  - Ensure namespace declarations are correct in the output XML

### Headers/Footers
- [ ] Implement `DocxConverter#convert_header(html, type)`:
  - Parse header.html HTML using the same paragraph conversion logic
  - Handle IE conditional comments (`<!--[if...]-->`, `<!--[endif]-->`)
  - Handle page number fields: `<span style='mso-field-code: PAGE'>` → `w:fldChar`/`w:instrText` run sequence
  - Return `Uniword::Wordprocessingml::Header` with paragraphs/tables

- [ ] Implement `DocxConverter#convert_footer(html, type)`:
  - Same as header but produces `Footer` objects
  - Handle the ISO pattern: two-column table with copyright text and page number

- [ ] Register headers/footers in `DocxPackage`:
  - Assign relationship IDs (rId) sequentially
  - Add content type overrides
  - Wire `HeaderReference`/`FooterReference` into section properties

### Sections
- [ ] Implement `DocxConverter#convert_sections(docxml)`:
  - Detect section breaks in the HTML (landscape divs, explicit page breaks)
  - Create `SectionProperties` for each section:
    - `PageSize` (A4: 11906x16838 twips)
    - `PageMargin` (from stylesheet or ISO defaults)
    - `HeaderReference`/`FooterReference` for each type (default, even, first)
    - `w:titlePg` for sections with first-page header/footer
    - `w:pgNumType w:start="1"` for body section
  - Handle mirror margins (`w:mirrorMargins`) from settings

- [ ] Map stylesheet `@page` rules to section properties:
  - Parse `size`, `margin`, `mso-header-margin`, `mso-footer-margin`
  - Handle landscape sections (swap width/height, adjust margins)
  - Map CSS `page:` property to section-specific `PageProperties`

## Key Files to Create/Modify
- `lib/html2doc/docx_converter.rb` — math, header/footer, section methods
- `lib/html2doc/base.rb` — provide header_file content to DocxConverter

## Success Criteria
- Math equations appear correctly in DOCX (inline and block)
- Headers and footers render as separate XML parts
- Multi-section documents have correct page geometry per section
- Page numbering resets correctly for body sections
- First-page and even-page headers/footers work independently
